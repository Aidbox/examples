CQL Engine + Aidbox

Evaluate CQL quality measures via Library/$evaluate on Aidbox with cqframework engine. Includes sample data for CMS130, CMS125, CMS131, CMS165.

01 Quick Start

Prerequisites: Docker with Docker Compose, curl.

Build and start the full stack:

bash 
docker compose up --build

This starts three containers: PostgreSQL, Aidbox (port 8888), and the CQL app (port 8080).

02 Verify Installation

Activate Aidbox — open http://localhost:8888 and follow the activation prompt.

Create a test patient via REST Console:

aidbox rest console 
POST /fhir/Patient
Content-Type: application/json
Accept: application/json

{
  "resourceType": "Patient",
  "gender": "male",
  "name": [
    {
      "family": "Smith"
    }
  ]
}

Expected: Status: 201 with Patient resource in response.

Evaluate example.cql:

aidbox rest console 
POST /Library/example/$evaluate

Expected: Status: 200

response 
{
  "resourceType" : "Parameters",
  "parameters" : [ {
    "name" : "MalePatients",
    "valueHumanName" : {
      "family" : "Smith"
    }
  } ]
}
Baseline confirmed. If you see the response above, the CQL engine is running and connected to Aidbox.

03 Load Shared Data

Before running CMS measures, load shared CodeSystems and stub resources (Organization + Practitioner).

Open a second terminal (keep the first one running with docker compose):

Navigate to the same directory where you ran docker compose up.

Load CodeSystems:

bash 
curl -u root:secret -X POST http://localhost:8888/fhir \
  -H "Content-Type: application/json" -d @data/codesystems-bundle.json

Create stub resources via REST Console (test data references these):

aidbox rest console 
POST /fhir
Content-Type: application/json

{
  "resourceType": "Bundle",
  "type": "transaction",
  "entry": [
    {
      "resource": {
        "resourceType": "Organization",
        "id": "example",
        "name": "Example Organization"
      },
      "request": { "method": "PUT", "url": "Organization/example" }
    },
    {
      "resource": {
        "resourceType": "Practitioner",
        "id": "example",
        "name": [{ "family": "Example", "given": ["Practitioner"] }]
      },
      "request": { "method": "PUT", "url": "Practitioner/example" }
    }
  ]
}

⚙ Architecture

Client → Aidbox → Spring Boot + CQL Engine → FHIR Data
POST /Library/{name}/$evaluate  →  custom operation routing  →  cqframework evaluate  →  FHIR Parameters response

dqm-content-qicore-2025 data structure

The dqm-content-qicore-2025 repository is the canonical source for CQL measures and test data. It contains:

PathContents
input/cql/91 standalone .cql files — all measures + shared libraries (FHIRHelpers, Hospice, Status, etc.)
input/tests/measure/{MeasureName}/Test data: one folder per patient, each containing individual FHIR resources as separate JSON files + a MeasureReport with the expected result (ip, den, exc, num)
bundles/measure/{MeasureName}/Pre-built transaction bundles combining everything: Library (with embedded CQL+ELM), Measure, ValueSets, all test patients, and expected MeasureReports

The data/ directory in this example contains bundles assembled from the same source with Aidbox-specific fixes applied (see Aidbox Adaptations).

Adding your own CQL library

  1. Place .cql file in src/main/resources/
  2. Rebuild: docker compose build cql-app && docker compose up -d cql-app
  3. Evaluate: POST /Library/{library-name}/$evaluate

⚠ Aidbox-Specific Adaptations

These adaptations bridge differences between Aidbox and HAPI FHIR Server (the reference test environment for cqframework):

AdaptationWhy
FullExpandTerminologyWrapperAidbox $expand may return incomplete results without explicit count parameter. Wrapper adds count=10000. HAPI FHIR returns all codes by default.
maxCodesPerQuery=64Large ValueSets cause HTTP 414 (URI Too Long) when codes are inlined in FHIR search URLs.
QICore data providerCMS measures use QICore profiles. Engine looks up data by model URI — without registering http://hl7.org/fhir/us/qicore, it silently returns no data.
kotlinx-io bridgeCQL engine 4.x (Kotlin) uses kotlinx.io.Source instead of java.io.InputStream for loading CQL files. Bridge code converts between the two.
buildResponse()CQL engine returns Java objects. This method serializes them into a FHIR Parameters JSON response.
BOX_FHIR_TERMINOLOGY_ENGINE: hybridAidbox requires explicit terminology engine configuration for $expand to work at all.
Stub Organization/example + Practitioner/exampleTest data Coverage and MedicationRequest resources reference these. Create both before loading clinical data.
ValueSet description + valueDate fixAidbox requires description (FHIR says optional) and rejects valueDate extensions.

⚠ Known Limitations

ToConcept(Quantity) bug — measures using USCoreBMIProfile fail with Could not resolve call to operator 'ToConcept'. Upstream cqframework issue #564, present in both 3.x and 4.x.

■ Shutdown

Run from the second terminal, from the same project directory where you ran docker compose up. Stop all containers (data preserved in Docker volume):

bash 
docker compose down

Or stop and delete all data (clean slate for next run):

bash 
docker compose down -v
